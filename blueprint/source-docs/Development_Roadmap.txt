
SINGAPORE HOME COOKS
Practical Development Roadmap
Every step from idea to fully developed product — Version 8.0 — Final
June 2026  |  Confidential
This roadmap is grounded in reality. Every small, obvious, and tedious step is included — not just the exciting ones. Challenges are identified honestly with resolutions proposed. Updated with sophisticated accounting system, three-tier dashboard, pain point solutions, frictionless SFA onboarding, and cook collaboration features.

1. Honest Capability Statement
1.1 What Claude Can Genuinely Do
	•	Write all frontend code: React/Next.js components, pages, forms, navigation, styling, responsive layouts, animations.
	•	Write all backend logic: Supabase database schema, API routes, authentication flows, business logic, data transformations.
	•	Build the complete double-entry accounting ledger: transaction posting, chart of accounts, reconciliation engine, all financial reports.
	•	Build every AI feature: calorie estimation, natural language meal search, AI dish descriptions, photo quality assessment, personalisation inference.
	•	Generate all automated workflows: order lifecycle, certificate expiry, payout calculation, email sequences, credit award triggers.
	•	Build all PDF generators: receipts, payout statements, corporate invoices, IRAS annual statements.
	•	Write all content: email templates, onboarding guides, Cook Agreement drafts, marketing copy.
	•	Debug code when things break — and things will break. This is normal development.

1.2 What Claude Cannot Do
	•	Deploy code to live servers independently — founder executes final deployment steps with guidance.
	•	Purchase domains, register accounts, sign contracts, or make payments.
	•	Guarantee zero bugs on first attempt — debugging is normal, not failure.
	•	Test on live devices independently — founder tests and reports back.
	•	Replace an accountant for SFRS-compliant statutory accounts.

1.3 What the Founder Must Contribute
	•	Active participation: be present to review, test, feedback, and decide. Weeks between sessions slow the build significantly.
	•	All account registrations: Supabase, Vercel, GitHub, Resend, Twilio, OneSignal, Xero, eventually HitPay.
	•	Testing on real phone and real browser — report exactly what is wrong, including exact error messages.
	•	Decision-making: when two approaches are possible, choose one and commit.
Reality check: The MVP can be built in 6–8 weeks of consistent work — meaning 3–4 sessions per week. If sessions happen once a fortnight, the same work takes 6–8 months of calendar time. Commitment to the build schedule is the single biggest factor in timeline.

2. Pre-Development Phase — Before Writing Code
2.1 Legal & Compliance (Weeks 1–3)
Step 1: Domain Registration
	•	Go to Vodien.com or Namecheap.com.
	•	Register singaporehomecooks.sg (requires Singapore entity/individual, ~S$30/year) and singaporehomecooks.com (~S$20/year).
	•	Do this today — domains can be taken while you wait.

Step 2: Trademark Filing
	•	Go to IPOS website (ipos.gov.sg). File for 'Singapore Home Cooks' under Class 43 (food and beverage services) and Class 42 (technology platform services).
	•	Cost: approximately S$240–340 per class. Budget S$500–700 total. File online — straightforward wordmark.
	•	Filing gives protection from the filing date even before mark is registered.

Step 3: Business Bank Account
	•	Open DBS Business Multi-Currency Account or OCBC Business Account.
	•	Register for PayNow Business — link to business UEN or mobile number.
	•	This account receives all customer payments and makes all cook payouts.

Step 4: Business Registration
	•	Register sole proprietor or private limited company with ACRA (acra.gov.sg).
	•	Sole proprietorship: S$115, simpler. Private limited: S$315, better for future investment.
	•	You need a UEN for the business bank account, PayNow, and corporate invoices.

Step 5: Legal Documents
	•	Engage Singapore food law specialist first (before Cook Agreement drafting) — formal written opinion on platform classification. S$2,000–3,000. Non-negotiable.
	•	Once opinion received: engage general commercial lawyer for Cook Agreement, Customer Terms, Privacy Policy.
	•	Brief precisely: 'We are a technology marketplace platform for independent SFA-registered home-based food businesses. We are not a food operator.'

Step 6: MAS Payment Services Act Confirmation
	•	Ask legal contact before first transaction: 'Does collecting PayNow payments and disbursing weekly to individual payees require a MAS Payment Services Act licence?'
	•	Document the answer in writing. This must be confirmed before any real money moves.

Step 7: Insurance Procurement
	•	Email NTUC Income, AXA Singapore, and MSIG Singapore simultaneously using exact description from Concept Document Section 16.
	•	Request: Combined Public and Product Liability (S$1M per claim, S$2M aggregate) + Cyber Liability + Directors and Officers.
	•	Allow 5–7 business days per response. Compare quotes. Purchase before first customer transaction.
	•	Budget: S$2,800–5,300/year total.

2.2 Accounts & Services Setup (Week 2)
Step 8: Create All Required Service Accounts
	•	Supabase (supabase.com): database, auth, file storage. Free tier. Project name: 'singapore-home-cooks'.
	•	Vercel (vercel.com): hosting. Free tier. Connect to GitHub.
	•	GitHub (github.com): code repository. Free. Private repo: 'shc-platform'.
	•	Resend (resend.com): email delivery. Free tier (100 emails/day). Verify your domain.
	•	Twilio (twilio.com): SMS. Pay-per-use (~S$0.05/SMS). Get Singapore sender number.
	•	OneSignal (onesignal.com): push notifications. Free tier.
	•	Xero (xero.com): accounting. S$30/month. Connect to business bank account via bank feed.
	•	Cloudinary (cloudinary.com): image and video storage. Free tier (25GB).
	•	Anthropic API (console.anthropic.com): AI features. Pay-per-use. Budget S$20–50/month at launch.

Step 9: Engage Accountant
	•	Engage Singapore accountant for initial Xero setup and configuration. S$500–800 one-time.
	•	Brief: sole proprietor or private limited company, marketplace business model, need Singapore tax treatment, GST tracking, and SFRS compliance.
	•	Annual accounting fees: S$1,500–3,000/year.

2.3 Design & Content Preparation (Weeks 2–3)
Step 10: Finalise Branding
	•	Brand colour confirmed: green #1D9E75. Commission logo — Fiverr, S$50–150 for clean wordmark.
	•	Collect 10–15 food photos for placeholder use during development — pilot cook visit photos ideal.

Step 11: Content Preparation
	•	Singapore food vocabulary synonym map: list 100+ dish name variation pairs for the search engine.
	•	10 most common FAQ answers for cooks and customers — feeds support chatbot.
	•	Hygiene disclaimer final text (after lawyer review).
	•	Cook onboarding email sequence: 5 emails (Days 1, 3, 7, 14, 30) — draft together with Claude.
	•	IRAS allowable expense categories list for cook expense tracker.

2.4 Manual Pilot (Weeks 1–6, Parallel)
Step 12: Run the Manual Pilot
	•	Recruit 5–10 cooks via Facebook groups and Instagram DMs.
	•	Create Instagram page @singaporehomecooks. Post cook photos and dishes.
	•	Take real orders via Instagram DM or WhatsApp. Collect PayNow to personal account.
	•	MUST include at least 3 occasion group orders above S$80 to validate AOV assumption.
	•	Document everything: what confuses cooks, what customers ask about, what goes wrong, what delights.
	•	Go/No-Go decision at end of pilot based on criteria in Concept Document Section 20.

4. Phase 1 Development — MVP (Weeks 1–8 of Build)
Build what enables a cook to list and a customer to find, pay, and collect. Build the accounting system, commission engine, agility foundations, and analytics schema from day one — retrofitting all of these is expensive.

4.1 Week 1: Authentication, Cook Registration, Accounting Foundation & Agility Setup
Step 16B: Commission Rules Engine
	•	Create commission_rules table: id, name, rule_type (enum), conditions (JSON), commission_rate, priority, effective_from, effective_to (nullable), is_active, created_by, notes.
	•	Seed with all current commission rules as database entries — do NOT hardcode any rate in application logic.
	•	Build evaluate_commission(order_value, cook_tier, cook_designation, occasion_type, customer_tier) function: queries active rules matching conditions, sorts by priority, returns lowest rate in cook's favour, applies S$2.00 floor.
	•	Every commission calculation in the platform calls this function — never uses hardcoded rates.
	•	CEO admin interface for commission rules: list all rules, create new rule, edit (creates versioned copy), deactivate, schedule future rule, preview calculator.

Step 16C: Platform Agility Foundations
	•	Create feature_flags table: feature_name, is_enabled, enabled_for (JSON), rollout_percentage, created_at, updated_at. Seed with all Phase 1 features set to enabled.
	•	Create platform_config table: key (unique), value (string), description, last_updated_by, last_updated_at. Seed with all configurable values.
	•	GST config keys to add at setup (all pre-registration values): gst_enabled: false, gst_rate: 0, gst_registration_number: null, gst_registered_from: null, gst_invoice_threshold: null, gst_display_mode: exclusive. These are already built in — flip gst_enabled to true when registration occurs.
	•	Build checkFeatureFlag(feature_name, userId) helper function used throughout codebase.
	•	Build getConfig(key) helper function — every configurable value fetched via this, never hardcoded.
	•	Register event bus: central event emitter. Define all event names. Each feature registers its listeners at application startup.
	•	CEO dashboard: Feature Flags panel (toggle any feature), Platform Config panel (edit any key-value).

Step 16D: Analytics Schema Setup
	•	Create analytics schema in Supabase separate from operational schema.
	•	Create core analytics tables: analytics.demand_gaps, analytics.cook_metrics, analytics.daily_summary, analytics.seasonal_patterns, analytics.customer_cohorts, analytics.geographic_data.
	•	Create customer_events table in operational schema: customer_id, event_type, metadata (JSON), created_at. Log every significant customer action from day one.
	•	Set up first nightly ETL job (Supabase scheduled function): daily_platform_summary — runs at 2am, calculates yesterday's GMV, orders, new cooks, new customers, cancellation rate, stores in analytics.daily_summary.
	•	Set up demand_gap_analysis job: runs nightly, finds searches with no orders by cuisine, dish, area, stores in analytics.demand_gaps. This feeds the 'Unmet demand near you' widget from Phase 2.

Step 16E: PDF Export Foundation
	•	Install and configure Puppeteer in the Next.js backend.
	•	Create base PDF HTML template: Singapore Home Cooks branding, header with logo and UEN, footer with page number and confidentiality notice.
	•	Build first PDF: per-order receipt. Test with a sample order. Verify formatting on both A4 and digital viewing.
	•	Build export API route: /api/export/[document_type]/[id] — accepts parameters, generates document, streams as file download.

Step 16F: ExcelJS Setup
	•	Install ExcelJS library.
	•	Build base Excel workbook helper: creates workbook with platform branding in header row, standard column widths, number formatting for currency (S$).
	•	Test with a simple single-tab export (daily earnings table). Verify formatting renders correctly in Excel and Google Sheets.

Step 17: Authentication System
	•	Login and signup pages for cook and customer accounts. Supabase Auth with email/password and Google OAuth.
	•	Role assignment at registration: cook or customer. Three additional roles in database only: operations, ceo (assigned manually by founder).
	•	Email verification required. Password reset flow. Session management (30-day login).

Step 18: Five-Stage Cook Onboarding Flow
	•	Stage 1 — Warm welcome: cook signs up, creates account immediately, starts building profile. SFA not mentioned yet. 'Welcome! Let us get your kitchen ready to earn.'
	•	Stage 2 — Profile building: display name, Cook Since year, bio, 'My Story' (free-form, no limit), cuisine specialties, area, profile photo.
	•	Stage 3 — SFA registration guide built into platform: step-by-step GoBusiness walkthrough. Platform opens GoBusiness page in new tab. Each step explained with WHY it benefits her. 'Your SFA registration number on your profile tells customers you are an approved food business.'
	•	Stage 4 — WSQ course: 'We pay for it. Here is how to enrol. SkillsFuture credits may cover it entirely for Singaporeans.'
	•	Stage 5 — Pending state: account created in pending_verification status. Profile complete, listings draftable, nothing live yet. Dashboard shows: 'Add your SFA registration number to go live.'
	•	Stage 6 — Go-live celebration: animated congratulations when both SFA number entered and WSQ certificate uploaded and verified by admin.

Step 19: Double-Entry Accounting Ledger Foundation
	•	Create ledger_entries table as append-only, immutable log. Add Supabase Row Level Security (RLS): only server-side functions can insert — no user can modify entries.
	•	Create chart_of_accounts table: load all accounts from chart of accounts in Concept Document Section 11.3.
	•	Build posting function: post_transaction(debit_account, debit_amount, credit_account, credit_amount, order_id, description). Validates that debit equals credit before inserting. Rejects unbalanced entries.
	•	Build reconciliation check function: compare sum of all ledger entries for account 2001 (Cook Earnings Payable) against sum of all unpaid payout records. Alert if discrepancy.

4.2 Week 2: Dish Listing Creation
Step 20: Interactive First-Listing Wizard
	•	For each cook's very first listing only: guided wizard that minimises inputs.
	•	Cook provides: ingredient list (structured entry), one photo, and a price.
	•	Platform automatically generates: AI dish description, allergen declaration pre-populated from ingredients, calorie estimate with confidence badge, price suggestion based on similar listings.
	•	Cook reviews all generated content and approves or edits. First listing in approximately 5 minutes.
	•	After first listing: full listing creation form available for subsequent listings.

Step 21: AI Dish Description Generator
	•	Structured ingredient entry form: ingredient name dropdown (Singapore ingredients pre-loaded), quantity field, unit dropdown (g, ml, cups, pieces, tbsp, tsp, 'to taste').
	•	'Generate Description' button: send to Claude API with prompt: 'You are helping a Singapore home cook write a professional listing description. Based on these ingredients: [list], write a 2–3 sentence description that is accurate, appetising, and culturally authentic to Singapore food. Include key flavours and notable characteristics. Do not invent ingredients not listed. Return only the description.'
	•	Display in editable text field — cook can modify before saving.

Step 22: AI Calorie Estimation
	•	On listing save: send ingredients to Claude API: 'Calculate estimated calories per portion. Ingredients: [list]. Number of portions: [n]. Cooking method: [method]. Return JSON only: {calories: integer, protein_ratio: float, carb_ratio: float, fat_ratio: float, confidence: full or category}'
	•	Parse response. If 5+ ingredients with quantities: confidence = full, green badge. Fewer: confidence = category, amber badge.
	•	Store in dish record. Display traffic light + macronutrient visual on listing page.

Step 23: AI Photo Quality Assessment
	•	On each dish photo upload: send image to Claude API vision model with prompt: 'Assess this food photo for quality. Provide exactly 3 specific actionable tips to improve it, each in under 15 words. Focus on: lighting, composition, background, colour, plating. Return JSON: {tip1: string, tip2: string, tip3: string}'
	•	Display tips as dismissible cards below the uploaded photo. Cook can dismiss and upload anyway — assessment is advisory.
	•	Cost: fraction of a cent per photo. Meaningful improvement in listing quality across entire platform.

Step 24: Full Listing Creation Form
	•	Dish name, category, cuisine type.
	•	Photo upload: up to 5 photos via Cloudinary. Video upload: optional, up to 60 seconds.
	•	Ingredient section with three-tier toggle: Ingredients Only / Add Technique Notes / Share Full Recipe.
	•	Allergen declaration: mandatory checkboxes — nuts/peanuts, shellfish/seafood, dairy, gluten/wheat, eggs, soy, halal/non-halal, may contain. Cannot submit without completing. Displayed ABOVE order button.
	•	Price per portion. Last-minute premium toggle: if enabled, premium % slider and advance window hours selector.
	•	Standard price and last-minute price shown in real-time earnings display: 'Standard price: S$12.00 → You receive: S$10.20. Last-minute price: S$13.80 → You receive: S$11.73'
	•	Availability type, daily portions limit, collection days and times.
	•	Dietary tags, occasion flags. SEO slug auto-generated from dish name and cook name.

Step 26: Occasion-First Homepage

	•	Schema.org markup on homepage for Singapore local food discovery.

Step 27: Dish Name Search with Synonym Map

4.4 Week 4: Order & Payment Flow
Step 28: Order Placement
	•	'Place Order' button creates order record in database with status pending_payment.
	•	Order number generated: SHC-[YYYY]-[MM]-[auto-increment padded to 5 digits].
	•	Order confirmation page: PayNow QR code (generated from business PayNow ID and exact amount), order summary, 30-minute countdown timer.

Step 29: Payment Confirmation — Phase 1 Manual Solution
	•	Customer taps 'I have paid via PayNow' button.
	•	Receipt number generated: RCP-[YYYY]-[MM]-[same sequential number as order].
	•	Post transaction to ledger: DR Cash (1001) full amount / CR Cook Earnings Payable (2001) cook net / CR Platform Commission Earned (3001) commission amount.
	•	Cook receives SMS and push notification immediately.
	•	Customer receives confirmation email with order number, receipt number, order summary.
	•	Cook's full address released to customer via email and order confirmation screen.
	•	In-app order chat thread opened automatically.
30-minute window: if customer does not confirm payment within 30 minutes, order is automatically cancelled and released. Cook is notified. Customer receives cancellation email with option to reorder.

Step 30: Cook Order Management Dashboard
	•	Incoming orders list with order number, dish, customer, quantity, collection date, status.
	•	Order action buttons: Confirm Order, Mark Ready for Collection, Mark Collected, Flag Issue.
	•	Availability toggle: prominent one-tap pause/unpause.
	•	Daily accounting view: all today's orders with gross, commission, net. Running total.

4.5 Week 5: Notifications & Automation
Step 31: Email Notification Templates
	•	Build all templates in Resend: order confirmation (customer), new order (cook), order confirmed by cook (customer), order ready (customer), order collected (both), review request (both — 2-hour delay), payout sent (cook), weekly earnings summary (cook), certificate expiry warnings (60 and 30 days), welcome email (cook and customer).

Step 32: Cook Onboarding Email Sequence
	•	Day 1: welcome, platform guide PDF, how to create first listing. Include link to first-listing wizard.
	•	Day 3: listing tips, photo quality assessment explanation, allergen declaration importance.
	•	Day 7: check-in, video upload prompt, demand calendar introduction.
	•	Day 14: one-week review, Academy intro, Cook Community invitation.
	•	Day 30: one-month milestone, pricing calculator intro, upcoming festive demand alert.

Step 33: Certificate Expiry Tracking
	•	Supabase cron job runs daily at 9am. Queries certificates expiring within 60 days.
	•	60-day threshold: send first reminder email. 30-day: send urgent reminder + SMS. Expired: set all cook listings to paused status, send suspension email with reactivation instructions.
	•	On new certificate upload and admin verification: reactivate all paused listings, send congratulations email.

4.6 Week 6: Address Privacy, Ratings & Reviews
Step 34: Address Privacy System
	•	Cook profile and listing pages: area only shown (e.g. 'Tampines').
	•	On order payment confirmed: release cook's full address to customer in confirmation email and order detail page only.
	•	Cook can set address_delay preference in dashboard: immediate (default) or 2 hours before collection time.
	•	In-app order chat opens automatically on payment confirmation. Cook and customer communicate without phone numbers.

Step 35: Automated Ratings & Reviews
	•	2-hour post-collection cron job: send review request to both cook and customer.
	•	Review form: 1–5 stars, optional written text (minimum 20 characters if written), optional photo.
	•	Review published automatically. Cook overall rating recalculated on each new review.
	•	48-hour reminder if no review submitted.

4.7 Week 7: Accounting System & Payouts
Step 36: Cook Accounting Dashboard
	•	Daily view: query ledger entries for today's cook's transactions. Display order number, dish, customer first name, gross, commission rate, commission amount, net. Running total. Export PDF/CSV.
	•	Weekly view: aggregate by week. Show payout reference when paid. Match calculated net against actual payout.
	•	Monthly view: aggregate by month with comparison chart. Export PDF/CSV.
	•	Annual IRAS statement (scheduled for 1 January each year): gross sales, total commission deducted, net income, cook's entered expenses, estimated taxable income. Downloadable PDF.

Step 37: Cook Expense Tracker
	•	Simple log: date, category (ingredients, packaging, gas/electricity, equipment, WSQ course, mobile phone, other), amount, optional receipt photo upload.
	•	Auto-totals by category and by month. Feeds into annual IRAS statement calculation.
	•	Pre-loaded help text for each category explaining IRAS allowability.

Step 38: Tax Estimate Calculator
	•	Takes: net platform income year-to-date + cook-entered expenses = estimated taxable income.
	•	Applies Singapore resident personal income tax rates (progressive scale).
	•	Shows estimated annual liability if current pace continues.
	•	Disclaimer: 'This is an estimate only. Consult a tax professional for your final filing.'
	•	Updates automatically as new orders complete.

Step 39: Three-Role Access Control
	•	Supabase RLS policies for three roles: cook, operations, ceo.
	•	Cook: can only see own records in all tables.
	•	Operations: can see all orders, cook accounts, customer issues. Cannot see CEO financial views.
	•	CEO: full access to all data including ledger, financial reports, analytics.
	•	Test every RLS policy exhaustively before launch. Cross-user data access must be completely blocked.

Step 40: Manual Payout Workflow
	•	Every Sunday evening: Supabase function calculates each cook's weekly earnings from ledger. Generates payout list for India assistant: cook names, PayNow IDs, net amounts.
	•	Monday morning: India assistant prepares transfer list, founder reviews and executes PayNow transfers.
	•	After each transfer: post to ledger: DR Cook Earnings Payable (2001) / CR Cash (1001). Mark payout as paid. Send automated earnings email to cook.
	•	Reconciliation check runs before payout execution: ledger balance vs payout sum vs bank balance. Must agree within S$0.01 or payout halted.

Step 41: Weekly Xero Export
	•	Every Monday: generate structured CSV of week's ledger entries in Xero-compatible format.
	•	Include: all income entries, commission entries, credit/refund entries, cook payout entries.
	•	Email to accountant for import into Xero. Two-minute process for accountant.
	•	This is the only connection between platform and Xero — no live API integration.

Step 41B: Phase 1 Receipt and PDF Export Suite
	•	RECEIPT NUMBER COUNTER: Create separate receipt_counter table distinct from order counter. Receipt counter increments only on confirmed payment — never on order placement. Annual reset 1 January. Format: RCP-YYYY-NNNNN.
	•	INDIVIDUAL CUSTOMER RECEIPT PDF: Build Puppeteer HTML template with all required blocks — header (logo, UEN, registered address), receipt ID block (receipt number, order number, date/time SGT, payment method), customer block, cook and dish block (including cook SFA number captured at transaction time), financial transparency block (amount to named cook, platform service fee % and amount, total, PayNow reference), footer with legally required platform facilitator language and GST statement.
	•	CORPORATE TAX INVOICE PDF: Extended template adding — document title TAX INVOICE or INVOICE, buyer details block (company name, UEN, billing address, attention), dietary group order breakdown table (halal/vegetarian/no-shellfish/standard portions with subtotals per category), bank transfer payment details, remittance advice slip (cut-off line).
	•	COOK PAYOUT STATEMENT PDF: Template with — payout reference, cook name and PayNow ID, period covered, detailed order table (every order: number, dish, collection date, gross, platform fee, net), PayNow transfer confirmation block, year-to-date running totals (gross, fees, net, order count, average order value).
	•	CREDIT NOTE PDF: Template for accounting corrections — credit note number (CN-YYYY-NNNNN), original receipt number reference, original amount, credit amount, reason, revised net amount. Note: original receipt is NEVER modified — credit note is the correction document.
	•	EMAIL DELIVERY: HTML receipt embedded inline in email (visible without downloading) + PDF attachment. Both contain identical information. Receipt resend available from customer order history at any time.
	•	RECEIPT STORAGE: Receipt data stored permanently in database for re-generation. Minimum 5-year retention per IRAS requirements. Cook SFA number stored at transaction time — not linked to current profile (ensures historical accuracy if SFA number changes).
	•	PRE-LAUNCH TEST: Generate test receipts for all four document types. Verify: all required fields present, commission split displayed correctly, legal footer language approved by lawyer, GST statement accurate, financial figures balance, PDF renders correctly on screen and when printed.

11. How We Work Together
How to Start a Development Session
	•	Always state which specific step number you want to work on. 'Let's build Step 22 — the search page' is far more productive than 'let's continue the platform.'
	•	Share the current state if something broke: copy the exact error message. Never paraphrase error messages — the exact text matters.
	•	Have VS Code open and your development server running (npm run dev) before the session starts.

What to Do With Code Provided
	•	Copy exactly into the specified file. Do not retype — copy and paste preserves exact formatting.
	•	After copying: save the file and check your browser. Tell what you see.
	•	If there is an error: copy the full error from the browser console (right-click > Inspect > Console tab) or terminal. Share in next message.
	•	If it works: confirm it works and we move to the next step.

When Things Break
	•	Things will break. This is normal development. Do not delete what was built — first understand what broke.
	•	Most common causes: typo in environment variable, missing import at top of file, database table not yet created, missing step.
	•	Share: exact error message, file where error occurs, last change made before error appeared.

Testing Protocol
	•	After each major feature: test yourself before moving on. Use the application as a real cook and as a real customer.
	•	Test on your phone — not just your laptop. Singapore Home Cooks will be used primarily on mobile.
	•	For accounting features: manually verify that ledger entries balance after each transaction type.
